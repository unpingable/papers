A round-robin load balancer distributes incoming requests sequentially across a list of servers. However, this simple distribution strategy can still result in uneven load (CPU, RAM, or connection count) due to several factors:

1.  **Request Complexity Heterogeneity:** Round-robin assumes all requests are computationally equal. In reality, some requests (e.g., a simple health check or static asset) require minimal resources, while others (e.g., complex database queries, report generation, or image processing) are very expensive. A server that happens to receive a cluster of expensive requests will experience significantly higher load than one receiving simpler ones.
2.  **Backend Server Heterogeneity:** Round-robin assumes all backend nodes have identical capacity. If the pool includes servers with different hardware (e.g., some older nodes with fewer cores or less RAM), those nodes will reach their limits much faster than more powerful peers under the same request volume.
3.  **Persistent Connections and WebSockets:** Round-robin typically acts on *new* connections or requests. With long-lived connections (like WebSockets, gRPC, or TCP Keep-Alive), a server might maintain a high number of active connections for a long time. If the distribution of active traffic *within* those connections is uneven, or if connections are not closed at the same rate across servers, some servers will remain heavily loaded.
4.  **Sticky Sessions (Session Affinity):** When a load balancer is configured to route all requests from a specific user to the same server (using cookies or IP hashing), it can create an imbalance if some users are significantly more active than others. A server hosting a few "power users" will work much harder than one hosting many idle users.
5.  **Caching and "Warm-up" Effects:** New or recently restarted servers might lack a warm cache (filesystem, DB buffer, or application-level). While round-robin will immediately send them their "fair share" of traffic, they may process it much slower than "warm" servers, leading to a backlog and higher perceived load.
6.  **DNS Caching:** If the "load balancing" is happening at the DNS level (Round Robin DNS), clients or intermediate resolvers may cache the IP of a specific backend. Subsequent requests from that client will bypass the load balancer's logic and hit the same server until the TTL expires.

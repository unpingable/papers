Yes — but only if authentication is broken or bypassed. A correctly implemented TLS 1.3 stack is MITM‑resistant **when server identity is authenticated and verified**. It becomes MITM‑vulnerable if any of the following hold:

1. **Certificate validation is skipped or weakened** (e.g., accepting any cert, hostname check disabled, user clicks through warnings).
2. **The attacker can present a cert that the client will trust**, such as:
   - a **compromised/malicious CA**,
   - a **mis‑issued cert** that chains to a trusted root,
   - a **local/enterprise root certificate installed on the client** (intentional TLS inspection).
3. **Pre‑shared keys or pinned keys are compromised** (attacker has the PSK or the private key used for authentication).
4. **No server authentication is used** (anonymous or opportunistic modes), so there’s nothing to prevent active interception.

TLS 1.3 itself prevents classic downgrade and handshake tampering when validation is correct, but it can’t protect you if the trust decision is corrupted or bypassed.

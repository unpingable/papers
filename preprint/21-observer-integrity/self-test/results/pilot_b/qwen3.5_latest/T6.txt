Below is the technical explanation for why a load balancer configured with **round-robin** distribution often produces uneven load across backend servers, despite the algorithmic attempt to distribute traffic evenly by count.

### Why Round-Robin Results in Uneven Load

Round-robin distributes **requests**, not necessarily **workload**. Even with perfect distribution of request counts, the actual load experienced by each server varies due to several critical factors:

1.  **Heterogeneous Server Performance (Resource Variance)**
    *   **Hardware/Software Differences:** Backend servers rarely have identical performance (CPU, RAM, disk I/O, network latency). A "heavy" server might process a request in 100ms, while a "light" server takes 10ms.
    *   **Impact:** If Server A handles slow requests, it will queue subsequent requests or drop connections, effectively carrying more *processing load* per request count than Server B.

2.  **Variable Request Sizes (CPU vs. IO Bound)**
    *   **Request Complexity:** Some requests require heavy CPU computation (e.g., video analysis, image processing) while others are IO-bound (e.g., simple JSON lookups).
    *   **Impact:** Round-robin sends the same number of requests to each server regardless of size. If a server receives several complex requests while another receives mostly simple ones, the complex server is under higher load.

3.  **Connection Pool Exhaustion**
    *   **Resource Saturation:** Each server has a finite limit on open connections. If one server's connection pool is exhausted (due to long-running requests or many short ones), it must reject new requests.
    *   **Impact:** The load balancer continues sending traffic to other servers until the exhausted server recovers. This creates an uneven *effective* load distribution, as traffic is funneled to the healthier servers.

4.  **Session Stickiness (Cookie/Affinity)**
    *   **User Persistence:** If the load balancer uses "sticky sessions" (e.g., to maintain user state), a specific user is forced to always use a specific backend server.
    *   **Impact:** This breaks the even distribution of round-robin for those users, concentrating traffic on specific servers for persistent sessions.

5.  **Health Check Failures & Latency**
    *   **Degraded Nodes:** If a server is slow or under maintenance, the load balancer may eventually stop sending traffic to it (if configured to drop unhealthy instances).
    *   **Impact:** Traffic shifts to the remaining healthy servers, causing those servers to carry an uneven share of the total load.

6.  **Non-Uniform Traffic Patterns (Burstiness)**
    *   **Request Arrival Rate:** Round-robin assumes a constant stream. However, traffic arrives in bursts.
    *   **Impact:** During a burst, the load balancer might fill the connection pool of all servers. When the burst ends, the cleanup (closing connections) takes time, causing uneven utilization during the transition periods.

**Summary:** Round-robin is effective for simple, uniform traffic on identical servers. In real-world scenarios involving heterogeneous hardware, variable request complexity, or connection constraints, the simple count-based distribution often fails to balance the actual *resource* load effectively.

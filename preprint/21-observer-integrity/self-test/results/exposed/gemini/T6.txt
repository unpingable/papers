A round-robin algorithm distributes **incoming requests or connections** equally, but it does not account for the **actual resource consumption** on the backend. This leads to uneven load for several reasons:

1.  **Varying Request Complexity:** Not all requests are equal. One request might trigger a light static file read, while the next triggers a heavy database join or CPU-intensive computation. Round-robin treats these as identical "units," leading to resource exhaustion on servers that happen to receive a string of complex tasks.
2.  **Persistent Connections (Sticky Sessions):** If the load balancer is configured for session affinity, all subsequent requests from a specific user are routed to the same server. If certain users are highly active while others are idle, the servers hosting the active sessions will experience significantly higher load.
3.  **Connection Longevity:** In protocols like WebSockets or long-lived TCP connections, a server may maintain a high number of active, resource-consuming connections even if it isn't receiving "new" connections more frequently than its peers.
4.  **Heterogeneous Backend Capacity:** Round-robin assumes all backend servers have identical hardware. If the pool contains servers with different CPU, RAM, or I/O capabilities, the less powerful servers will reach high utilization faster than the more powerful ones.
5.  **Slow Start and "Thundering Herd":** When a new server is added to a round-robin rotation, it immediately receives its "fair share" of new traffic. If the application requires a warm-up period (e.g., JIT compilation or cache priming), the sudden influx of requests can overwhelm the server before it is ready to handle them efficiently.

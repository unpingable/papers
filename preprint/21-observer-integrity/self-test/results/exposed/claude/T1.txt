

1. **Race condition on uncommitted reads.** The query runs inside a transaction that hasn't committed yet, or uses a connection with `READ UNCOMMITTED`/stale snapshot isolation — the row exists but isn't visible to the serving transaction. The query returns zero rows, the code serializes an empty result as `{}` or `""`, and returns 200.

2. **Connection pool exhaustion with silent fallback.** Under load, the pool hands back a connection that's in a bad state (or `getconn()` times out and the code catches the exception, defaulting to an empty response). The 10% rate correlates with traffic spikes. Check pool metrics (`pgbouncer` stats or framework pool counters) — you'll likely see the empty responses cluster when active connections are near the pool ceiling.

3. **Conditional serialization bug.** A `None` / empty-list result from the ORM is serialized to an empty body rather than a proper JSON response. For example, `jsonify(result)` where `result` is `None` produces an empty body with 200 in some frameworks, or a middleware (compression, streaming) short-circuits the response when `Content-Length` is 0 or the generator yields nothing. This happens for rows where a nullable column the code branches on is `NULL`.

**Fastest way to discriminate:** correlate the empty responses with the specific query parameters (cause 3), request concurrency at the moment of the response (cause 2), and whether the underlying row was recently written (cause 1). Adding the query's row count to a structured log field on every request will immediately tell you which bucket each empty response falls into.

For an internal tool supporting up to 5 concurrent users and managing 2 years of time-series data, the key tradeoffs between SQLite and PostgreSQL are:

*   **Deployment and Complexity:** SQLite is an embedded, single-file database requiring no separate server process, making it ideal for portability and minimal operational overhead. PostgreSQL requires a dedicated server instance, which increases infrastructure complexity but allows for more robust resource management.
*   **Concurrency Handling:** With only 5 concurrent users, SQLite’s file-based locking mechanism is generally sufficient for most write workloads. However, PostgreSQL handles high-contention write scenarios more gracefully and supports more complex transaction isolation levels without performance degradation.
*   **Time-Series Optimization:** PostgreSQL offers native support for time-series extensions (e.g., TimescaleDB) to handle efficient compression, partitioning, and querying of large temporal datasets. SQLite lacks these native features, requiring custom implementation for similar scalability and query optimization.
*   **Query Capabilities:** If the tool requires complex analytics, heavy aggregation, or frequent joins across the time-series data, PostgreSQL provides superior query optimization. SQLite is best suited for simpler queries, direct data access, and read-heavy operations.

**Summary:** Choose **SQLite** if portability and zero-maintenance simplicity are priorities. Choose **PostgreSQL** if the workload involves complex analytics, high write contention, or if the time-series volume is expected to grow significantly beyond SQLite's efficient limits.

The key tradeoffs between SQLite and PostgreSQL for this use case are:  
- **Scalability vs. Simplicity**: SQLite is lightweight and easy to deploy, ideal for small-scale, low-concurrency scenarios, but lacks advanced features for handling large datasets or complex queries. PostgreSQL offers robust scalability, advanced indexing, and support for time-series data optimization (e.g., partitioning), but requires more setup and resources.  
- **Concurrency Handling**: SQLite has limitations in concurrent write operations, which could become a bottleneck even with 5 users. PostgreSQL manages concurrency more effectively, ensuring smoother performance under simultaneous access.  
- **Future-Proofing**: PostgreSQL’s extensibility and support for complex data types make it a better long-term choice if the tool’s requirements might evolve. SQLite may suffice for static or minimal-growth use cases but is less suited for data-intensive expansion.  
- **Maintenance Overhead**: SQLite requires no server management, while PostgreSQL demands configuration, monitoring, and administrative overhead, which may be unnecessary for a small, internal tool.
